# Phase 2 Drift Scoring — Empirical Validation Results

> Generated by `eval-drift.mjs` on 2026-04-11.
> Purpose: Establish whether Phase 2's proposed drift scoring algorithm
> (originally "plain Jaccard over N=M=3 windows, threshold 0.3") actually
> detects topic drift reliably in realistic LLM coding sessions.
>
> This document is the evidence base for Phase 2's final algorithm choice.
> All conclusions in `PHASE2_SPEC.md` regarding algorithm selection, default
> threshold, extended stopwords, and the 2-consecutive-turn rule trace back
> to the measurements recorded here.

## Motivation

During design review, a concern was raised: *"if the detector cannot
reliably detect when a new topic enters the session, this tool is pointless."*
The original spec had converged on plain Jaccard similarity over symmetric
N=M=3 keyword sliding windows with a default threshold of 0.3, justified
by theoretical considerations (simplicity, constraint-satisfaction,
literature precedent). No empirical measurement had been performed against
realistic coding-session data.

This validation addresses that gap. The goal is not to prove the design
optimal, but to verify that it is **usable** and to identify concrete
improvements if it is not.

## Corpus

A hand-written corpus of **15 scenarios** covering six categories, designed
to probe each of the theoretical failure modes identified during design
review. Each scenario is a list of 8–10 realistic user messages in the
voice of a developer working with an LLM coding assistant.

| Category          | Count | Expected behavior                               |
| ----------------- | ----- | ----------------------------------------------- |
| `clean_shift`     | 4     | Detector SHOULD fire — unambiguous topic change |
| `no_drift`        | 3     | Detector should NOT fire — stable single topic  |
| `gradual`         | 3     | Detector SHOULD fire — slow multi-turn shift    |
| `generic_masking` | 2     | Detector SHOULD fire despite generic vocabulary |
| `synonymy`        | 2     | Detector should NOT fire on restated same topic |
| `tangent_return`  | 1     | Detector should NOT fire on brief digression    |

The corpus deliberately includes one Korean-language `clean_shift` scenario
to verify that the cross-language tokenizer from Phase 1 handles Hangul
correctly.

The corpus is stored inline in `eval-drift.mjs` for reproducibility.
Scenarios are not drawn from real user data — they are synthesized to
exhibit the failure-mode patterns in a controlled way. The tradeoff
(synthetic vs. real) is discussed in the Caveats section below.

## Variants Tested

Six scoring variants were evaluated across a threshold sweep of
`[0.05, 0.10, 0.15, 0.20, 0.25, 0.30, 0.40, 0.50, 0.60, 0.70, 0.80]`:

| Variant | Tokenizer      | Decision rule                                   |
| ------- | -------------- | ----------------------------------------------- |
| Plain   | Phase 1        | Single-turn `score < threshold`                 |
| Path A  | Phase 1 + ext. stopwords + stemming | Single-turn `score < threshold` |
| Path B  | Path A         | Session-local IDF weighted Jaccard, single turn |
| Path C  | Path A         | **Two consecutive turns both `< threshold`**    |
| Path D  | Path A         | Rolling mean of last 2 turns `< threshold`      |
| Path E  | Path A         | Rolling mean of last 3 turns `< threshold`      |

The **extended stopwords** list in Path A adds approximately 80 generic
coding-domain terms (`file`, `function`, `fix`, `test`, `run`, `use`,
`method`, `class`, etc.) on top of Phase 1's base English stopword list.
These terms were chosen because they appear across topic events of almost
any coding session and pollute set-overlap calculations.

The **stemmer** in Path A is a lightweight Porter-inspired rule set
(approximately 50 lines) that strips the most common English suffixes
(`ing`, `ed`, `er`, `tion`, `ment`, `ly`, `s`, etc.). It is applied only
to ASCII tokens so Hangul is unaffected.

## Results

### Best achievable F1 per variant

| Variant                                   | Best threshold | Precision | Recall | **F1** |
| ----------------------------------------- | -------------- | --------- | ------ | ------ |
| Plain Jaccard (Phase 1 stopwords only)    | 0.10           | 0.727     | 0.889  | 0.800  |
| Path A (extended stopwords + stemming)    | 0.10           | 0.692     | 1.000  | 0.818  |
| Path B (Path A + session-local IDF)       | 0.05           | 0.727     | 0.889  | 0.800  |
| **Path C (Path A + 2-consecutive rule)**  | **0.10**       | **0.818** | **1.000** | **0.900** |
| Path D (Path A + 2-turn rolling mean)     | 0.10           | 0.818     | 1.000  | 0.900  |
| Path E (Path A + 3-turn rolling mean)     | 0.10           | 0.818     | 1.000  | 0.900  |

### Min-score separation at the default (Path A)

The minimum Jaccard score observed across all turns of each scenario,
grouped by drift-expected vs. no-drift-expected:

| Group                         | Min   | Mean  | Max   |
| ----------------------------- | ----- | ----- | ----- |
| Drift-expected scenarios (9)  | 0.000 | 0.018 | 0.059 |
| No-drift scenarios (6)        | 0.038 | 0.075 | 0.130 |

**Separation gap** (no-drift min − drift max) = `-0.021`. This is a
**negative** separation gap, meaning no single-turn threshold can
perfectly separate the two groups. This is the empirical origin of the
F1 ≤ 0.82 ceiling observed for single-turn rules (Plain, Path A, Path B).

### Key insight from per-turn score traces

Dumping the per-turn scores for scenarios whose minimum score lies in the
ambiguous zone reveals that **stable-topic scenarios exhibit isolated,
one-turn dips in Jaccard score**. For example:

```
S04 [no_drift] React shopping cart:
  t5=0.16  t6=0.25  t7=0.15  t8=0.04  t9=0.08
```

The single dip at `t8=0.04` is below any practical single-turn threshold,
but the surrounding turns are clearly within the stable range. Drift
scenarios, in contrast, exhibit **sustained** low scores across multiple
consecutive turns:

```
S01 [clean_shift] React UI → PostgreSQL schema:
  t5=0.09  t6=0.07  t7=0.03  t8=0.00

S02 [clean_shift] Auth bug → README docs:
  t5=0.11  t6=0.03  t7=0.03
```

This temporal structure is **mechanically exploitable** by a decision
rule that requires two consecutive turns below threshold, which is
exactly what Paths C, D, and E implement.

## Decision

**Phase 2 will ship with: Path A tokenizer + 2-consecutive-turn rule +
default threshold `0.10`.**

Rationale:

1. **Recall = 1.000.** The winning configuration catches every drift in
   the corpus. This is the single most important property per the
   original design brief ("if the detector cannot detect, this tool is
   pointless").
2. **Precision = 0.818** (2 false positives out of 6 no-drift scenarios)
   is acceptable under topic-fence's "fence, not wall" philosophy — false
   positives manifest as ignorable warnings, not destructive actions.
3. **F1 = 0.900** is a substantial improvement over plain Jaccard's
   empirical ceiling of F1 = 0.800. The 10-percentage-point gain is
   attributable to two orthogonal improvements: extended stopwords
   (Path A) adds ~2 points of recall, and the consecutive-turn rule
   (Path C) adds ~8 points of precision.
4. **Path C is stateless in practice.** Although "two consecutive turns"
   sounds temporal, it can be computed statelessly at each scorer call
   by fetching 6 historical topic events (instead of 5) and comparing
   two adjacent window pairs. No persisted state is required.
5. **Paths D and E are equivalent to C on this corpus** but introduce
   additional arithmetic (rolling mean) without improving the metric.
   C is chosen for its straightforward interpretation: "fire only when
   low-overlap persists".
6. **Path B (session-local IDF) did not improve over plain Path A.**
   The theoretical hope — that down-weighting frequent keywords would
   sharpen drift detection — did not materialize on this corpus. IDF
   remains a legitimate candidate for Phase 4 re-evaluation with a
   larger corpus, but should not be added to Phase 2.

## Production Implications

### Algorithm change

The scorer now computes **two** Jaccard scores per invocation: one for
the window pair `[events[0..3], events[3..6]]` and one for
`[events[1..4], events[4..7]]`. Firing requires both scores to fall
below the threshold.

```
history = last 6 topic events (not 5)
current = topic event from the current message
combined = [...history, current]  // 7 events total

prevOld = combined[0..3]           // events 0, 1, 2
prevNew = combined[3..6]           // events 3, 4, 5
prevScore = jaccard(union(prevOld), union(prevNew))

currOld = combined[1..4]           // events 1, 2, 3
currNew = combined[4..7]           // events 4, 5, 6
currScore = jaccard(union(currOld), union(currNew))

if (prevScore < THRESHOLD && currScore < THRESHOLD) fire drift event
```

### Cold start

Cold start requires 7 topic events (6 historical + 1 current), up from
the original spec's 6. Under default window sizes this means the first
possible drift firing is on the 7th topic-bearing user turn, which is
acceptable for realistic session lengths.

### Configuration

The default threshold changes from `0.3` to `0.10`. The environment
variable `CONTEXT_MODE_TOPIC_DRIFT_THRESHOLD` remains the override
mechanism. Clamping range `[0.0, 1.0]` is unchanged.

### Extended stopwords

Approximately 80 generic English terms are added to the base Phase 1
stopword list. The full list is inlined in `topic-fence.ts` and should
be regarded as a design artifact to be tuned against real data in
Phase 4. The additions are all common coding-domain filler words whose
removal did not harm any drift-expected scenario in the corpus.

### Stemmer

A small Porter-inspired stemmer (~50 lines) is added to `topic-fence.ts`.
It handles the most common English suffixes and is applied only to ASCII
tokens. Hangul and other non-ASCII tokens pass through unchanged. The
stemmer is internal to the tokenizer and is not exposed as a public API.

## Caveats

1. **Corpus size.** 15 scenarios is a small sample. Confidence intervals
   on the F1 numbers are wide. The 0.082 F1 gap between Path A and Path C
   is based on classifier outcomes changing on only 2 scenarios; any
   individual scenario that behaved differently could shift the result.
   **However, the mechanism underlying the improvement (consecutive-turn
   noise filtering) is mechanically explainable and expected to transfer.**
2. **Synthetic scenarios.** The corpus was hand-written, not drawn from
   real sessions. Synthesis introduces two biases: (a) scenarios may be
   more "clean" than real conversations, making drift easier to detect;
   (b) the author's intuitions about what a drift "looks like" are baked
   into the ground-truth labels. Phase 4 should re-validate against real
   captured topic-event streams once those exist.
3. **Ground-truth labels are scenario-level, not turn-level.** Scenarios
   are labeled "drift expected" or "no drift expected" at the scenario
   level. The detector could fire at the "wrong" turn within a drift
   scenario and still be credited with a true positive. This is a
   deliberate simplification.
4. **Threshold overfitting.** The best threshold (0.10) was selected from
   an 11-point sweep. With 15 scenarios, selecting any specific threshold
   is susceptible to overfitting. The production default `0.10` should be
   treated as a reasonable starting point, not a proven optimum. Phase 4
   should re-tune from real data via ROC analysis.
5. **Path A stopword list is not principled.** The extended stopword list
   was hand-curated based on intuition about "generic coding vocabulary".
   A data-driven approach (e.g., computing session-local term frequency
   distributions and automatically down-weighting high-frequency terms)
   was considered (Path B) but did not improve over Path A on this corpus.
   Phase 4 should investigate whether a principled derivation would
   transfer better than hand-curation.

## Phase 4 Follow-up Items

These items are explicitly deferred to Phase 4, driven by real data
captured after Phase 2 ships:

1. Re-tune `TOPIC_DRIFT_THRESHOLD`, `TOPIC_WINDOW_OLD`, and
   `TOPIC_WINDOW_NEW` against real captured topic-event streams.
2. Re-evaluate Path B (session-local IDF) against the larger real-data
   corpus. The theoretical motivation remains sound; it simply did not
   manifest measurable benefit on the small synthetic corpus.
3. Re-evaluate the extended stopword list — consider whether a subset
   could be removed without harming recall, or whether additional
   terms should be added.
4. Re-evaluate the stemmer — consider whether specific suffix rules
   are net-helpful or net-harmful in production.
5. Consider whether cross-session vocabulary statistics could be
   aggregated into a per-user (not cross-user) IDF model, bridging
   the gap between session-local IDF (tested here) and full TF-IDF
   (rejected for architectural reasons).
